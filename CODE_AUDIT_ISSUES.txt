Fixing the `chat-media`/`documents` "public bucket listing" advisory findings
(Section 13 context) by simply DROPPING their public SELECT policies broke
uploads to both buckets entirely: Postgres RLS requires `INSERT ... RETURNING`
to pass a SELECT policy for the new row to be returned, and Supabase
Storage's upload endpoint always does a RETURNING-style insert. With zero
SELECT policy, every upload — KYC verification, rider documents, chat
photos/voice notes — failed with "new row violates row-level security
policy for table \"objects\"", confirmed by a live user report and

finding has NOT regressed.

bucket's SELECT policy down to zero — if broad/public read isn't wanted,
replace it with a scoped one in the same migration, don't just drop it.

1. CRITICAL — money / privilege-escalation bugs (fix before launch)
================================================================================

1.1 complete_delivery() lets anyone redirect a rider's payout to any wallet
    File: supabase/migrations/20260707000000_launch_readiness_fixes.sql:239-290
    The RPC takes `p_rider_user_id` as a plain parameter and calls
    `_wallet_credit(p_rider_user_id, price - commission, 'payout', ...)` without ever
    checking `p_rider_user_id = auth.uid()` or that this user is the order's assigned
    rider. Any authenticated user who learns/guesses an order's 4-digit delivery OTP
    can call this RPC with someone else's wallet id (or their own) and redirect the
    payout — a direct funds-diversion bug, and it was introduced by the migration
    that was supposed to be the "launch readiness fix".
    Fix: look up the real rider from `orders.rider_id -> riders.user_id` inside the
    function and use that value for the credit; ignore/validate `p_rider_user_id`
    against `auth.uid()`.

1.2 claim_order() lets any user assign any order to any rider
    File: supabase/migrations/20260519200000_claim_order_rpc.sql:2-35
    SECURITY DEFINER function never checks `p_user_id = auth.uid()` nor that
    `p_rider_id` belongs to that caller. Any authenticated caller can pass an
    arbitrary rider id and hijack a `searching_rider` order, and can forge the
    `order_events.created_by` audit trail with an arbitrary user id.
    Fix: add
      IF p_user_id <> auth.uid()
         OR NOT EXISTS (SELECT 1 FROM riders WHERE id = p_rider_id AND user_id = p_user_id)
      THEN RETURN false; END IF;
    at the top of the function body.

1.3 orders UPDATE policy lets customers/riders rewrite price, status, rider_id directly
    File: supabase/migrations/20260707000000_launch_readiness_fixes.sql:78-84
    (re-declared from 20260517154310_...sql:229-233)
    The `order_update` policy only restricts *which rows* can be touched (ownership
    check via USING), not *which columns/values* may change. Because the app relies
    on RPCs (`claim_order`, `complete_delivery`) for the "correct" state machine, but
    the table itself has no column-level restriction, any customer or rider can call
    `supabase.from('orders').update({ status: 'delivered', price_xof: 1 })` directly
    from the browser and bypass the RPC flow entirely — free deliveries, self-marking
    orders complete without an OTP, or rewriting the fare after booking.
    Fix: add a `WITH CHECK` that pins the immutable columns (e.g. compare
    `price_xof`/`commission_xof`/`customer_id` against the pre-update row via a
    trigger, since RLS WITH CHECK can't reference OLD) — practically, add a
    `BEFORE UPDATE` trigger that rejects changes to `price_xof`, `commission_xof`,
    `status` unless performed by the RPCs (`SECURITY DEFINER`, checked via
    `current_setting`) or an admin/dispatcher role.

1.4 wallet_transactions has RLS enabled but no INSERT policy — top-up flow is broken
    File: supabase/migrations/20260517154310_cd981ceb-bf06-4467-a5bf-e7f30be7f62d.sql:216
    (only `tx_own_select` exists; grepped all migrations, no INSERT policy anywhere)
    src/routes/_authenticated/app.wallet.tsx `confirmTopup()` does a client-side
    `.insert()` into this table to record "I sent the transfer, please review it" —
    this will be rejected by RLS for every non-admin user in production. The
    top-up-confirmation feature is currently non-functional and fails silently as a
    generic toast error.
    Fix: add `CREATE POLICY tx_own_insert ON wallet_transactions FOR INSERT TO
    authenticated WITH CHECK (auth.uid() = user_id AND status = 'pending')`, or move
    the insert into a dedicated RPC.

1.5 rate_delivery() accepts a client-supplied customer id — rating spoofing
    File: supabase/migrations/20260519300000_delivery_features.sql
    Takes `p_customer_id` as a parameter instead of using `auth.uid()`. Any
    authenticated user who knows an order id and the real customer's id can submit
    or overwrite that customer's rating for a rider.
    Fix: drop the parameter and use `auth.uid()` inside the function body.

1.6 dispatcher assignment uses a bare, non-atomic UPDATE (race condition)
    File: src/routes/dispatcher/index.tsx:54-65
    `assignRider()` does `orders.update({ rider_id, status: 'rider_assigned' })
    .eq('id', orderId)` with no `status = 'searching_rider'` precondition. Two
    dispatchers (or the same dispatcher double-clicking) assigning the same order
    concurrently can both succeed, silently overwriting each other —
    double-assignment. The rider-side flow (src/routes/rider/dispatch.tsx:71-81)
    already solves this correctly via the atomic `claim_order` RPC.
    Fix: route dispatcher assignment through `claim_order` (once 1.2 is fixed) or an
    equivalent RPC with a `status='searching_rider'` guard, instead of a bare update.

1.7 [STATUS: appears RESOLVED — verified 2026-07-20, see Section 8 note] Merchant API keys stored in plaintext
    Files: supabase/migrations/20260518000000_rider_admin_merchant.sql,
           20260520000000_merchant_api_keys.sql
    Column is named `key_hash TEXT` with a comment saying "store bcrypt hash in
    production," but no hashing is ever applied — the raw key is stored as-is. The
    later "launch readiness" migration's own header explicitly calls this out as a
    known problem but never fixes it. (RLS ownership on the table itself is fine.)
    Fix: hash keys with bcrypt/argon2 before insert (in the edge function or an RPC
    that generates the key), store only the hash + a short displayable prefix, and
    rotate any keys already issued in plaintext.

2. HIGH — authorization gaps in edge functions
================================================================================

2.1 agora-token function mints call tokens without checking channel membership
    File: supabase/functions/agora-token/index.ts
    Verifies the caller has a valid JWT and applies rate limiting, but never checks
    that the user is actually a participant of the conversation the channel
    represents. `channelName` is derived client-side as `call-${conversationId}`
    (src/hooks/use-agora-call.ts:29) and only regex-validated server-side. Any
    authenticated user who knows/guesses a conversationId can obtain a PUBLISHER
    token and join or disrupt someone else's call.
    Fix: before minting, parse the conversationId out of channelName and confirm the
    caller is one of the two participants in `conversations`, 403 otherwise.

2.2 translate function allows overwriting arbitrary message rows
    File: supabase/functions/translate/index.ts:~103-108
    Uses the service-role client to `.update(...).eq('id', messageId)` where
    `messageId` comes straight from the request body, with no check that the caller
    is part of the conversation that message belongs to. Any authenticated user can
    overwrite `translated_content`/`translate_from` on any message by id.
    Fix: join against conversation membership in the same query
    (`select ... where id = messageId and conversation_id in (caller's conversations)`)
    before allowing the update.

2.3 CORS allowlist only contains localhost — production origin is not configured
    File: supabase/functions/_shared/cors.ts:5-11
    `ALLOWED_ORIGINS` lists dev URLs only; the real deployed origin lines are
    commented out, so requests from the live frontend fall through to a hardcoded
    fallback (`https://rapide.africa`) that won't match the actual Vercel/production
    domain unless that's the final custom domain already live. If it isn't, calls to
    agora-token/translate are broken in production right now.
    Fix: add the actual deployed origin(s) to `ALLOWED_ORIGINS` before/at each
    deploy, ideally driven by an env var rather than a hardcoded list.

2.4 Rate limiter has a check-then-insert race
    File: supabase/functions/_shared/rateLimiter.ts:13-25
    `checkRateLimit` reads the current count and inserts the new attempt as two
    separate calls, so concurrent requests from the same user can all read the same
    pre-insert count and all pass — the limit can be bypassed by firing requests in
    parallel. (It is correctly DB-backed, not in-memory, so it does survive across
    cold starts/instances — only the atomicity is the problem.)
    Fix: wrap the check+insert in a single RPC/transaction, or use a Postgres
    advisory lock / `INSERT ... ON CONFLICT` counter pattern.

FIX 1.1 — complete_delivery() payout redirection
--------------------------------------------------------------------------------
New migration file, e.g. supabase/migrations/20260718000000_fix_complete_delivery.sql:

  CREATE OR REPLACE FUNCTION public.complete_delivery(
    p_order_id uuid,
    p_otp      text
  )
  RETURNS boolean
  LANGUAGE plpgsql
  SECURITY DEFINER
  SET search_path = public
  AS $$
  DECLARE
    v_otp         text;
    v_status      text;
    v_price       bigint;
    v_commission  bigint;
    v_rider_id    uuid;
    v_rider_user  uuid;
    rows_affected integer;
  BEGIN
    SELECT o.delivery_otp, o.status, o.price_xof, o.commission_xof, o.rider_id
    INTO   v_otp, v_status, v_price, v_commission, v_rider_id
    FROM   public.orders o
    WHERE  o.id = p_order_id;

    IF NOT FOUND                        THEN RETURN false; END IF;
    IF v_status != 'in_transit'         THEN RETURN false; END IF;
    IF v_otp IS NULL OR v_otp != p_otp  THEN RETURN false; END IF;

    -- Derive the payee from the order's assigned rider — never trust a
    -- client-supplied wallet/user id for a money-moving operation.
    SELECT user_id INTO v_rider_user FROM public.riders WHERE id = v_rider_id;
    IF v_rider_user IS NULL THEN RETURN false; END IF;

    -- Only the assigned rider (or an admin, for support overrides) may confirm.
    IF v_rider_user != auth.uid() AND NOT public.has_role(auth.uid(), 'admin') THEN
      RETURN false;
    END IF;

    UPDATE public.orders
    SET    status = 'delivered', delivered_at = now()
    WHERE  id = p_order_id AND status = 'in_transit';
    GET DIAGNOSTICS rows_affected = ROW_COUNT;

    IF rows_affected != 1 THEN RETURN false; END IF;

    INSERT INTO public.order_events (order_id, status, created_by)
    VALUES (p_order_id, 'delivered', v_rider_user);

    UPDATE public.riders SET total_deliveries = total_deliveries + 1
    WHERE id = v_rider_id;

    PERFORM public._wallet_credit(
      v_rider_user,
      GREATEST(v_price - v_commission, 0),
      'payout',
      NULL,
      p_order_id,
      'Delivery payout'
    );
    RETURN true;
  END;
  $$;

  GRANT EXECUTE ON FUNCTION public.complete_delivery(uuid, text) TO authenticated;

  - Wherever `complete_delivery` is invoked (search for it in

FIX 1.2 + 1.6 — claim_order() hijacking, and dispatcher's non-atomic assignment
--------------------------------------------------------------------------------
Same migration file as above, or a new one. Rewrite claim_order to (a) derive the
caller from auth.uid() instead of trusting p_user_id, and (b) support two valid
callers: the rider themself, or a dispatcher/admin assigning on a rider's behalf
(this also fixes 1.6, since dispatcher/index.tsx should route through this RPC):

  CREATE OR REPLACE FUNCTION public.claim_order(
    p_order_id uuid,
    p_rider_id uuid
  )
  RETURNS boolean
  LANGUAGE plpgsql
  SECURITY DEFINER
  SET search_path = public
  AS $$
  DECLARE
    rows_affected integer;
    v_is_self     boolean;
  BEGIN
    SELECT EXISTS (
      SELECT 1 FROM public.riders WHERE id = p_rider_id AND user_id = auth.uid()
    ) INTO v_is_self;

    IF NOT (
      v_is_self
      OR public.has_role(auth.uid(), 'dispatcher')
      OR public.has_role(auth.uid(), 'admin')
    ) THEN
      RETURN false;
    END IF;

    UPDATE public.orders
    SET    rider_id = p_rider_id, status = 'rider_assigned'
    WHERE  id = p_order_id AND status = 'searching_rider' AND rider_id IS NULL;

    GET DIAGNOSTICS rows_affected = ROW_COUNT;

    IF rows_affected = 1 THEN
      INSERT INTO public.order_events (order_id, status, created_by)
      VALUES (p_order_id, 'rider_assigned', auth.uid());
      RETURN true;
    END IF;
    RETURN false;
  END;
  $$;

FIX 1.3 — orders UPDATE policy allows rewriting price/status directly
--------------------------------------------------------------------------------
This one needs a product decision before it can be applied blindly: customers
legitimately need to change `status` to `cancelled` in some flow (check
src/routes/_authenticated/app.orders.tsx and app.track.$orderId.tsx for a client
cancel button before writing the trigger). Starting point — a BEFORE UPDATE
trigger that allows admins/dispatchers to change anything, and allows customers
exactly one self-service transition (cancel from a pre-dispatch state), but
blocks every other protected-column change for non-privileged callers:

  CREATE OR REPLACE FUNCTION public.enforce_order_update_guard()
  RETURNS trigger
  LANGUAGE plpgsql
  SECURITY DEFINER
  SET search_path = public
  AS $$
  BEGIN
    IF public.has_role(auth.uid(), 'admin') OR public.has_role(auth.uid(), 'dispatcher') THEN
      RETURN NEW;
    END IF;

    -- Customer self-service cancel: only status may change, only from a
    -- pre-dispatch state, only by the order's own customer.
    IF auth.uid() = OLD.customer_id
       AND OLD.status IN ('pending', 'searching_rider')
       AND NEW.status = 'cancelled'
       AND NEW.price_xof      IS NOT DISTINCT FROM OLD.price_xof
       AND NEW.commission_xof IS NOT DISTINCT FROM OLD.commission_xof
       AND NEW.rider_id       IS NOT DISTINCT FROM OLD.rider_id THEN
      RETURN NEW;
    END IF;

    -- Everything else: money/state fields are immutable for non-privileged callers.
    IF NEW.price_xof      IS DISTINCT FROM OLD.price_xof
       OR NEW.commission_xof IS DISTINCT FROM OLD.commission_xof
       OR NEW.status         IS DISTINCT FROM OLD.status
       OR NEW.rider_id       IS DISTINCT FROM OLD.rider_id
       OR NEW.customer_id    IS DISTINCT FROM OLD.customer_id THEN
      RAISE EXCEPTION 'Direct modification of protected order fields is not allowed';
    END IF;

    RETURN NEW;
  END;
  $$;

  CREATE TRIGGER trg_enforce_order_update_guard
  BEFORE UPDATE ON public.orders
  FOR EACH ROW EXECUTE FUNCTION public.enforce_order_update_guard();

Before shipping this, audit every client-side `.from("orders").update(...)` call
(book/track/rider/dispatcher routes) to confirm the only non-privileged update
path left is the cancel case above — anything else needs its own RPC the way
claim_order/complete_delivery already work.

FIX 1.7 — merchant API keys stored in plaintext
--------------------------------------------------------------------------------
Requires a new edge function (or extending an existing merchant-key-issuing one)
since hashing must happen before the row is written, and the raw key can only be
shown once. Recommended shape:

  1. Client calls a new `merchant-api-keys/create` edge function (service-role,
     not a direct table insert) with the merchant id.
  2. The function generates a random key: `crypto.randomUUID() +
     crypto.randomUUID()` (or `crypto.getRandomValues`), keeps a short prefix
     (e.g. first 8 chars) for display, and stores a SHA-256 hash of the full key
     via Web Crypto (`crypto.subtle.digest("SHA-256", ...)`, available natively
     in Deno — no extra dependency needed):
       const raw = cryptoRandomKey();                 // e.g. "rpd_live_<32 hex>"
       const hash = toHex(await crypto.subtle.digest("SHA-256",
         new TextEncoder().encode(raw)));
       await supabaseAdmin.from("merchant_api_keys").insert({
         merchant_id, key_hash: hash, key_prefix: raw.slice(0, 12), is_active: true,
       });
       return new Response(JSON.stringify({ key: raw }));   // shown once only
  3. On each authenticated API call using a merchant key, hash the presented key
     the same way and look up by `key_hash`, never by comparing raw strings.
  4. Rotate/revoke any keys already issued in plaintext under the current scheme
     — they must be treated as already compromised once this ships.

FIX 2.4 — rate limiter check-then-insert race
--------------------------------------------------------------------------------
File: supabase/functions/_shared/rateLimiter.ts. Move the check+insert into a
single SQL function so it's atomic under concurrent calls:

  CREATE OR REPLACE FUNCTION public.check_and_log_rate_limit(
    p_user_action text,
    p_max_requests integer,
    p_window_seconds integer
  ) RETURNS boolean
  LANGUAGE plpgsql SECURITY DEFINER SET search_path = public
  AS $$
  DECLARE v_count integer;
  BEGIN
    LOCK TABLE public.rate_limit_log IN SHARE ROW EXCLUSIVE MODE;
    SELECT count(*) INTO v_count FROM public.rate_limit_log
    WHERE user_action = p_user_action
      AND created_at >= now() - (p_window_seconds || ' seconds')::interval;

    IF v_count >= p_max_requests THEN RETURN false; END IF;

    INSERT INTO public.rate_limit_log (user_action) VALUES (p_user_action);
    RETURN true;
  END;
  $$;

Then rateLimiter.ts becomes a single RPC call:

  export async function checkRateLimit(supabase, userId, action, maxRequests, windowSeconds) {

      p_user_action: `${userId}:${action}`,
      p_max_requests: maxRequests,
      p_window_seconds: windowSeconds,
    });
    return { allowed: !!allowed, remaining: undefined };
  }

12.11 CRITICAL BUG FOUND WHILE SCOPING THIS, FIXED SAME PASS: every order
      ever booked was invisible to every rider
      Not in the original audit. All three insert paths (app.book.tsx,
      app.errand.tsx, merchant/bulk.tsx) create orders with status='pending'.
      But every rider-matching path — the dispatch queue
      (rider/dispatch.tsx, including its realtime filter
      `status=eq.searching_rider`), claim_order (`WHERE status =
      'searching_rider'`), and the dispatcher assignment view — only ever

      order that existed in production was stuck at 'pending', unclaimable,

12.12 SELF-CAUGHT REGRESSION DURING THIS PASS: expire_stale_orders had no
      access control
      Unlike every other RPC added in this pass, expire_stale_orders has no
      internal auth.uid()/has_role() check — by design, since it's only
      meant to run from the pg_cron job. But Postgres grants EXECUTE to
      PUBLIC by default on CREATE FUNCTION, and Supabase exposes every
      public-schema function to both `anon` and `authenticated` via
      PostgREST regardless of that — so until revoked, any signed-in (or
      even anonymous) caller could hit

12.2 "Payment Pending" / "Confirmed" imply a payment gateway that doesn't exist
     Every order today is cash-on-delivery — confirmed via the
     `complete_delivery` migration's own comment ("Assumes cash-on-delivery
     ... since there is no payment gateway integrated yet and no
     payment-method selector in the booking flow"). Adding a Payment Pending
     state ahead of Confirmed only makes sense once there's a real prepaid
     path (wallet-prepay or card/mobile-money gateway) that can actually put
     an order in a "waiting on payment" state. Open question: is payment
     gateway integration in scope alongside this redesign, or should
     Payment Pending/Confirmed be modeled as a no-op pass-through for COD
     orders (auto-advance immediately) until a real gateway exists?

12.3 "Rider Accepted" as distinct from "Rider Assigned" changes the claim flow
     Today, for the rider self-serve flow, `claim_order` IS the accept
     action — assignment and acceptance are the same atomic transition (see
     Section 1 FIX 1.2). For dispatcher-assigned orders (Section 9), the
     order also currently goes straight to rider_assigned with no separate
     "rider must confirm" step. Splitting these means: dispatcher-assigned
     orders would sit in "Rider Assigned" until the rider explicitly taps
     something to move to "Rider Accepted" (a new rider-side action, ties
     into 9.5's missing Reassign — a rider timing out at "Rider Assigned"
     without accepting is exactly when Reassign would trigger). Self-claimed
     orders could collapse both transitions into one atomic action as today.

12.4 "Delivery Verification" as its own status vs. today's OTP modal
     Today OTP entry is a client-side modal at status `in_transit`; the
     `complete_delivery` RPC checks the OTP and jumps straight from
     in_transit to delivered in one atomic step (Section 1 FIX 1.1). Making
     Delivery Verification a real intermediate status means splitting that
     RPC into two steps (enter verification state, then confirm) — worth
     confirming this is wanted, since the current atomic version is exactly
     what prevents a half-verified order from being left in a weird state if
     a rider's connection drops mid-flow.

12.5 "Delivered" vs "Completed" — what's the actual boundary?
     Needs a product decision on what separates them: candidates are (a)
     customer has rated the delivery, (b) the rider's payout has been
     credited (already instant today per 1.1/7.8, so this would be
     immediate), or (c) a dispute/refund window has closed. Whichever it is
     determines whether Completed is set by a trigger, a cron job, or stays
     manual.

12.6 "Expired" needs a real TTL job — today there's only a passive warning
     Section 9's "Needs Attention" panel on the admin ops dashboard
     (admin/index.tsx, `stuckOrders` query) already flags orders stuck in
     searching_rider >10 minutes, but it's a read-only alert with a manual
     Cancel button — nothing auto-transitions an order to any terminal state
     on a timer. "Expired" would need a scheduled job (pg_cron or an edge
     function on a timer) that actually flips old unclaimed orders.

12.7 "Rejected" needs the rider-decline tracking that doesn't exist yet
     Directly depends on Section 7 finding 7.3: the dispatch queue's
     "reject" (the X button) only dismisses the popup locally today — it
     never touches the server. A real Rejected status requires that decline
     to actually be recorded (and decide: does one rider rejecting move the
     whole order to Rejected, or just remove that rider as a candidate and
     keep the order in Awaiting Rider for others? Almost certainly the
     latter — "Rejected" as an order-terminal state probably means something
     closer to "customer's request was rejected" (e.g., by an admin/merchant
     review step), not "a rider declined it," which needs disambiguating).

12.8 "Refunded" as an order-level status vs. today's transaction-level refund
     `refund` already exists as a `wallet_transactions.type` value (used
     freely in admin/finance.tsx's GMV/refunds KPI), but there's no
     corresponding order-level status — an order stays "delivered" or
     "cancelled" forever even if it was later refunded. Adding Refunded to
     `order_status` means either the order can leave its terminal state
     (unusual — most systems keep the delivery outcome status and add a
     separate `refund_status` column instead) or Refunded needs to be
     modeled as an overlay/flag rather than a mutually-exclusive enum value.
     Worth deciding: separate `orders.refund_status` column vs. folding it
     into `order_status`.

Grepped every route for `.channel(` / `postgres_changes` vs `refetchInterval`

18.1 [MISSING] Dispatcher dashboard is 100% polling, zero realtime — despite
     being explicitly named in the spec

20.2 Optimistic updates: TanStack Query's actual primitive (`setQueryData`
     inside `onMutate`) was used in zero files before this pass
     Converted two real mutations to the full onMutate/onError-rollback/
     onSettled pattern:
       - rider/index.tsx's `toggleOnline`: flipping online/offline now
         updates the cached rider row instantly instead of waiting for the
         round-trip + invalidate + refetch; rolls back to the previous
         cached value if the update fails.
       - app.notifications.tsx's `markRead`/`markAllRead`: marking a
         notification read (or all of them) updates both the
         `notifications` list cache and the `notif-unread` badge count
         instantly, with rollback on failure. Previously these were a bare
         async call + invalidate (markRead wasn't even a `useMutation`, just
         a plain async function with no pending/error state at all).

20.3 The pattern to reuse going forward (not a new abstraction — just the
     shape both conversions above follow)
     Forms: `useForm<T>({ resolver: zodResolver(schema), defaultValues })`
     + `<Form {...form}><form onSubmit={form.handleSubmit(mutate)}>` +
     `<FormField control={form.control} name="x" render={({field}) => ...} />`.
     Keep zod schemas returning non-nullable field types with sensible empty
     defaults (e.g. `{address:"",lat:0,lng:0}` rather than `null` for an
     object field) — a `.nullable()` field on the schema fights zodResolver's
     generic typing in the installed @hookform/resolvers version (hit and
     fixed during the errand.tsx conversion).
     Mutations: `onMutate` cancels in-flight queries for the affected
     key(s), snapshots the previous cache value, writes the optimistic
     value via `setQueryData`, and returns `{ previous }`; `onError` restores
     it from context; `onSettled` invalidates for eventual consistency with
     the server regardless of outcome.
